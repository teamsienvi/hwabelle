Amazon SP-API Setup — Standard Operating Procedure (SOP)

Purpose: Step-by-step instructions for setting up Amazon Selling Partner API (SP-API) access so we can automate Multi-Channel Fulfillment (MCF) orders (or other API actions) from a connected application.
Who does this: The primary Seller Central account holder (the client).
Time estimate: ~15-30 minutes (plus Amazon review period of 1-3 business days).

What You'll Be Creating

By the end of this guide, you will have generated three credentials that we need for the integration:

Credential What It Is
Client ID Identifies your application to Amazon (like a username)
Client Secret A private key paired with the Client ID (like a password)
Refresh Token A long-lived token that lets our system make API calls on your behalf
⚠️ CAUTION: Treat these credentials like passwords. Never share them over unsecured channels. We recommend sharing via a secure method (e.g., a password manager share link or OneTimeSecret).

Phase 1 — Create a Solution Provider Portal (SPP) Account

ℹ️ NOTE: Amazon is migrating all developer management to the Solution Provider Portal. If you have already registered as a developer in Seller Central, you can still sign in here.
  1. Go to the Solution Provider Portal: https://developer.amazonservices.com
  2. Click Sign In or Create Account.
  3. Sign in using your primary Amazon Seller Central credentials (the main email/password you use to log into Seller Central).
  4. If prompted, complete two-factor authentication (2FA).

Phase 2 — Complete Your Developer Profile

This registers your Amazon seller account as a "Private Developer" — meaning you're building an app that only your own business will use. There is no fee for private developer registration.

  1. Once signed into the Solution Provider Portal, go to Settings → Developer Profile (or click the prompt to "Register as a Developer").
  2. Fill out the Developer Profile form:
    • Organization name: Your legal business entity name.
    • Contact email & phone: Your primary business details.
  3. Data Access: From the dropdown, select: "Private Developer: I build application(s) that integrate my own company with Amazon Services APIs"
  4. Roles: Select the roles that apply to the integration. For an MCF fulfillment integration, ensure these are selected:
    • ✅ Direct-to-Consumer Shipping
    • ✅ Amazon Fulfillment
  5. Use Cases (Template): Note the exact functionality being developed. You may copy and paste the generalized statement below or modify it as needed:
    "We are setting up a private integration between our own sales channels/websites and Amazon Multi-Channel Fulfillment (MCF). The system will use the Fulfillment Outbound API to automatically submit incoming orders to our FBA inventory for picking, packing, and shipping directly to the customer, as well as the Orders API to update tracking statuses."
  6. Security Controls (Template): Describe how your company protects Amazon credentials. You may copy and paste the generalized statement below or modify it to match your infrastructure requirements:
    "All SP-API credentials (LWA tokens, AWS keys) are stored purely as encrypted environment variables in our secure cloud platform. Credentials are never exposed in client-side code, frontend applications, or public source control repositories. Development access is strictly limited to authorized engineering personnel using role-based access control."
  7. Check the box confirming you agree to the Solution Provider Agreement, Acceptable Use Policy, and Data Protection Policy.
  8. Click Register.
  9. Wait for Amazon's review (typically 1-3 business days). If Amazon asks follow-up questions via case log/email, respond within 5 days to prevent case closure.

Phase 3 — Register Your Application

Once your developer profile is approved, you can create your private app directly in the portal.

  1. Sign in to the Solution Provider Portal.
  2. Click Add new app client (or "Register an application").
  3. Fill in the application details:
    • App name: e.g., "Internal E-Commerce Sync" or "MCF Integration"
    • API Type: SP-API
    • App type: Private
  4. IAM ARN: Amazon requires an AWS IAM Role ARN securely tied to the Application. Paste the ARN generated for you by our team (for example: arn:aws:iam::123456789012:role/SellingPartnerAPIRole).
  5. Click Save and Exit or Register.
ℹ️ NOTE: Your application will be in "Draft" status. This is perfectly fine — private apps do not need to be published. You can self-authorize a draft app.

Phase 4 — Self-Authorize Your Application

This is the critical step where you generate the Refresh Token.

⚠️ IMPORTANT: To self-authorize a Seller Central account, you must be the Primary User of that account. Secondary users cannot self-authorize.
  1. Find your newly created app in the list on the Solution Provider Portal.
  2. Look for the "Authorize app" button and click it.
  3. A Refresh Token will appear on the screen.
  4. Copy this token immediately and save it securely! This is your one chance to see it displayed.

Phase 5 — Retrieve Your Client ID and Client Secret

  1. In the Solution Provider Portal, find your application and click Edit app (or "View").
  2. Click View under "LWA credentials" for your application.
  3. You will see:
    • Client Identifier (Client ID) — starts with amzn1.application-oa2-client.
    • Client Secret — a long 64-character alphanumeric string (Ensure you copy all 64 characters!)
  4. Copy both values.

Phase 6 — Send Us the Credentials

Once you have all three values, please share them with us securely:

  • 1. Client ID (amzn1.application-oa2-client.abc...)
  • 2. Client Secret (amzn1.oa2-cs.v1.abc...) (Ensure nothing is cut off!)
  • 3. Refresh Token (Atzr|IQEBLzAtAh...)

Recommended Secure Sharing Methods:

  • ✅ OneTimeSecret.com — Create a secret link that self-destructs after being read once.
  • ✅ Password manager share — Use a service like 1Password or LastPass.
  • ❌ Do NOT send these in a regular email, text message, or chat.
